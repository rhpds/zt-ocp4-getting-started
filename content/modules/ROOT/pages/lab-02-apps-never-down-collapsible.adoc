= Module 2: Apps That Never Go Down
:source-highlighter: rouge
:toc: macro
:toclevels: 2

The DataSphere datacenter visualization is already running in your `{user}-datasphere` project.
In this module, you'll explore what keeps it running when things go wrong -- by breaking it on
purpose and watching OpenShift respond automatically. Then you'll configure those same resilience
features from scratch in your own project.

== Learning objectives

By the end of this module, you'll be able to:

* Explain how ReplicaSets maintain pod availability
* Configure a Horizontal Pod Autoscaler (HPA) for automatic scaling
* Describe the difference between liveness and readiness probes
* Demonstrate zero-downtime resilience under pod failure and traffic load

== Section 1: What keeps your app running?

The pre-deployed DataSphere app has resilience features already configured: a ReplicaSet,

=== Step 1.3: Auto-scaling under load

You just scaled manually. That works, but it requires you to be watching. What if traffic
spikes at 3am?

The Horizontal Pod Autoscaler (HPA) watches CPU metrics and automatically adjusts the pod
count. When CPU climbs above a threshold, it adds pods. When load drops, it removes them.
The pre-deployed DataSphere API already has an HPA configured.

[%collapsible]
.Using the Terminal (CLI)
====
Check the current HPA:

[source,role="execute"]
----
oc get hpa
----

[%collapsible]
.Expected output
=====
----
NAME             REFERENCE                   TARGETS        MINPODS   MAXPODS   REPLICAS   AGE
datasphere-api   Deployment/datasphere-api   cpu: 12%/50%   1         5         1          15m
----
=====

The `TARGETS` column shows current CPU utilization versus the 50% threshold that triggers
scaling. The following command combines several views into a live dashboard -- run it now,
then drag the DataSphere *Simulated Traffic* slider to about *50%*:

[source,role="execute"]
----
watch -n 2 "echo '=== Auto-Scaler ==='; oc get hpa; echo; echo '=== Pod CPU ==='; oc adm top pods --selector component=api; echo; echo '=== Pods ==='; oc get pods --selector component=api"
----

NOTE: Make sure the DataSphere slider is at 0% before starting. If the API has already
scaled up from earlier activity, the HPA numbers will not reflect a clean baseline.

=== Step 1.4: Health probe: detect a sick pod

What if a pod is still running but broken? A crashed pod gets replaced by the ReplicaSet,
but a sick pod that's alive and consuming normal CPU needs a different mechanism: the liveness probe.

The DataSphere API has a liveness probe on the `/health` endpoint. If it fails 3 times in a
row, OpenShift restarts the container. A readiness probe on the same endpoint removes the pod
from the Service load balancer the moment it stops passing -- so no traffic reaches the broken pod.

Trigger a health failure. Switch to the *Terminal* tab and run:

[source,role="execute",subs="attributes+"]
----
curl -s -X POST https://$(oc get route datasphere-ui -o jsonpath='{.spec.host}')/api/break
----

[%collapsible]
.Expected output
====
----
{"status":"broken","message":"Health check will now fail. OCP will restart this pod."}
----
====

NOTE: This request hit 1 of the API pods through the Service load balancer. Only that pod
has its health flag set to broken -- the others remain healthy.

[%collapsible]
.Using the Terminal (CLI)
====
Watch the pods:

[source,role="execute"]
----
watch oc get pods --selector component=api
----

Within 30 seconds, the *RESTARTS* column increments on one pod. Press `Ctrl+C` to stop.
====

[%collapsible]
.Using the Web Console
====
Go to *Workloads* → *Pods* and watch the `datasphere-api-*` pods. Within about 40 seconds:

. One pod shows `0/1` in the *Ready* column: the readiness probe pulled it from the Service
. The *Restarts* counter increments: the liveness probe restarted the container

// SCREENSHOT NEEDED: Workloads → Pods showing datasphere-api pods — one pod with Restarts = 1 (or showing 0/1 Ready transitioning), others still Running
image::console-guide/lab-02-apps-never-down/11-pods-api-probe-restart.png[Pods list showing one datasphere-api pod with Restarts incremented to 1]

TIP: Click the restarted API pod name and open the *Events* tab to see the liveness probe
failure events that triggered the restart.
====

[NOTE]
====
Notice the difference from Step 1.1: deleting a pod removes it entirely and the ReplicaSet
creates a replacement. Here, the liveness probe *restarts* the existing container in place.
Probes catch sick applications; ReplicaSets catch dead ones.
====

=== Step 2.4: Add health probes

The HPA handles capacity. Health probes handle a different failure mode: a pod that's alive
and consuming normal CPU, but silently broken. Without probes, that pod keeps receiving
traffic even though every request fails.

Two probe types work together here:

* *Liveness probe*: asks "is this container alive?" -- if it fails enough times, OpenShift restarts it
* *Readiness probe*: asks "can this container handle traffic right now?" -- if it fails, the Service stops sending requests to that pod

[%collapsible]
.Using the Terminal (CLI)
====
Add a liveness probe:

A few settings worth knowing:

* *`initialDelaySeconds`*: How long to wait before the first check. If your app takes 30
  seconds to start up, set this to 35 -- too short and the probe restarts the app before
  it finishes booting, creating a restart loop.

* *`failureThreshold`*: How many consecutive failures before action. Setting 1 is too
  aggressive; setting 10 is too forgiving. 3 is a reasonable default.

* *`timeoutSeconds`*: How long to wait for a response before counting it as a failure.
  The default is 1 second. Increase this for health endpoints that call downstream services.

== Module summary

Imagine running 500 services in production. Pods crash at 3am. Traffic spikes every Friday
afternoon. A memory leak quietly degrades one pod in a ten-pod deployment.

With what you configured today, none of that requires a human:

* The *ReplicaSet* detects the crashed pod and creates a replacement
* The *HPA* watches CPU climb and adds capacity before users notice slowness
* The *liveness probe* catches the degraded pod and restarts it before the error rate budges

[cols="1,2",options="header"]
|===
| What you did | How it works

| Deleted a pod (Section 1)
| The ReplicaSet detected the missing pod and created a replacement automatically

| Scaled to 3 replicas (Section 1)
| Losing 1 pod of 3 leaves 2 still serving traffic -- the Service routes around the failure

| Watched auto-scaling (Section 1)
| The HPA measured CPU against resource requests and added pods when load crossed the threshold

| Triggered a health probe failure (Section 1 and 2)
| The liveness probe detected the broken pod and restarted it; the readiness probe kept traffic away during recovery

| Configured all three features from scratch (Section 2)
| Resource limits → HPA → liveness probe → readiness probe -- the same four steps that production workloads follow
|===
