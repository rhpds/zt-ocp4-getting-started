= Module 2: 3 Ways to Deploy an Application
:source-highlighter: rouge
:toc: macro
:toclevels: 2

There is no single right way to get an application running on OpenShift.
In this module you will deploy *DataSphere* — a 2-tier operations dashboard — using 3 different methods:
from a pre-built container image, from source code using Source-to-Image, and as a packaged Helm chart.

Each method has its place, and by the end you will know when to reach for each one.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Deploy an application from a pre-built container image using `oc new-app`
* Trigger a Source-to-Image build from a Git repository
* Deploy a multi-component application in a single command using a Helm chart
* Understand when each deployment method is the right choice

== Section 2: Explore the architecture (5 minutes)

DataSphere has 2 components that work together.

=== The frontend: datasphere-ui

The UI is a React application served by Nginx. It was deployed directly from a pre-built container image:

[source,role="execute"]
----
oc describe deployment datasphere-ui | grep -A2 "Containers:"
----

[%collapsible]
.Expected output (key fields)
====
----
  Containers:
   datasphere-ui:
    Image:      quay.io/rhdp/datasphere-ui:1.0.0
----
====

The image is pulled from a container registry and run directly.
No source code was compiled. No build was needed.

=== The backend: datasphere-api

The API is a Python FastAPI application. It handles all data requests from the frontend:

[source,role="execute"]
----
oc describe deployment datasphere-api | grep -A2 "Containers:"
----

[%collapsible]
.Expected output (key fields)
====
----
  Containers:
   datasphere-api:
    Image:      quay.io/rhdp/datasphere-api:1.0.0
----
====

In this pre-deployed version, the API runs from a pre-built image.
In Section 3, you will build it from source code instead.

=== How they connect

The UI container runs Nginx, which forwards any request starting with `/api/` to the
`datasphere-api` Service at port 8080.

[source,role="execute"]
----
oc get service datasphere-api
----

[%collapsible]
.Expected output
====
----
NAME             TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)    AGE
datasphere-api   ClusterIP   172.30.14.22   <none>        8080/TCP   5m
----
====

Notice that `datasphere-api` has no Route: it's only reachable from within the cluster.
The UI talks to it by Service name. OpenShift's internal DNS resolves that name to the correct
pod automatically, even as pods restart and change IP addresses.

[mermaid]
....
flowchart LR
    B["🌐 Browser"] -->|"HTTPS"| R["Route\n(datasphere-ui)"]
    R --> UI["datasphere-ui\n(React + Nginx)"]
    UI -->|"/api/* → datasphere-api:8080"| API["datasphere-api\n(Python FastAPI)"]
....

=== Step 3.1: Deploy the UI from a container image

The UI image is already built and published. Deploy it directly:

[source,role="execute"]

oc new-app quay.io/rhdp/datasphere-ui:1.0.0 --name=datasphere-ui
----

[%collapsible]

     'oc expose service/datasphere-ui'

`oc new-app` created 3 resources in one command: an ImageStream to track the image,
a Deployment to manage the pods, and a Service for internal network access.

=== Step 3.2: Build the API from source code

The API source code lives in a Git repository.
Instead of building a container image yourself, you point OpenShift at the repository
and it handles the rest.

This is *Source-to-Image* (S2I): OpenShift detects the language, installs dependencies,
and packages the result into a container image — automatically, with no Dockerfile required.

[source,role="execute",subs="attributes+"]
----
oc new-app python~{datasphere_api_repo} --name=datasphere-api
----

[%collapsible]
.Expected output
====
----
--> Found image ... (Python 3.11 on UBI 9)
    ...
--> Creating resources ...
    imagestream.image.openshift.io "python" created
    imagestream.image.openshift.io "datasphere-api" created
    buildconfig.build.openshift.io "datasphere-api" created
    deployment.apps "datasphere-api" created
    service "datasphere-api" created
--> Success
    Build scheduled, use 'oc logs -f buildconfig/datasphere-api' to track its progress.
----
====

Notice the `buildconfig` in the output: that's the S2I build definition.
OpenShift is now cloning the repository, installing Python dependencies, and packaging
the result into a container image stored in the cluster's internal registry.

Watch the build:

[source,role="execute"]
----
oc logs -f buildconfig/datasphere-api
----

[%collapsible]
.Expected output (truncated)
====
----
Cloning "{datasphere_api_repo}" ...
Generating dockerfile with builder image image-registry.openshift-image-registry.svc:5000/openshift/python:latest
STEP 1/9: FROM image-registry.openshift-image-registry.svc:5000/openshift/python:latest
...
Collecting fastapi
Collecting uvicorn
...
Successfully installed fastapi-0.115.0 uvicorn-0.31.0 ...
...
Pushing image image-registry.openshift-image-registry.svc:5000/{user}-lab/datasphere-api:latest ...
Push successful
----
====

Once the build finishes, OpenShift automatically deploys the newly built image.
Wait for the rollout:

[source,role="execute"]
----
oc rollout status deployment/datasphere-api
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-api" successfully rolled out
----
====

== Section 4: Deploy the full stack as a Helm chart (5 minutes)

You deployed DataSphere manually in a handful of commands across 2 steps.
Helm lets you package all of those resources into a single installable unit.

The DataSphere Helm chart defines the same Deployments, Services, and Route you just created — but
bundled so the entire application can be installed, upgraded, or removed as one operation.

Create a new Project for this deployment:

[source,role="execute",subs="attributes+"]
----
oc new-project {user}-datasphere-helm
----

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
Now using project "{user}-datasphere-helm" on server "{openshift_api_url}".
----
====

The DataSphere Helm chart is pre-staged in your home directory. Install it:

[source,role="execute"]
----
helm install datasphere ~/datasphere-gitops
----

[%collapsible]
.Expected output
====
----
NAME: datasphere
LAST DEPLOYED: ...
NAMESPACE: {user}-datasphere-helm
STATUS: deployed
REVISION: 1
NOTES:
DataSphere deployed. Get the UI URL with:
  oc get route datasphere-ui -o jsonpath='{.spec.host}'
----
====

Wait for both pods to come up:

[source,role="execute"]
----
oc rollout status deployment/datasphere-ui && oc rollout status deployment/datasphere-api
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-ui" successfully rolled out
deployment "datasphere-api" successfully rolled out
----
====

Open the application:

[source,role="execute"]
----
oc get route datasphere-ui -o jsonpath='{.spec.host}'
----

The same 2-tier application, fully operational, from a single `helm install` command.

When you're done, remove the entire application in one step:

[source,role="execute"]
----
helm uninstall datasphere
----

[%collapsible]
.Expected output
====
----
release "datasphere" uninstalled
----
====

One command to install everything. One command to remove everything.

== Module summary

You deployed the same 2-tier application using 3 different methods.

*What you accomplished:*

* Deployed a pre-built container image with `oc new-app`
* Built a Python application from source using Source-to-Image — no Dockerfile required
* Deployed the complete stack with `helm install`
* Observed how OpenShift's internal DNS connects components automatically by Service name

*The 3 deployment methods:*

* *Container image*: fastest path when the image is already built and published
* *Source-to-Image*: build directly from a Git repository; OpenShift handles the container build
* *Helm chart*: package the entire application for repeatable, version-controlled deployments
