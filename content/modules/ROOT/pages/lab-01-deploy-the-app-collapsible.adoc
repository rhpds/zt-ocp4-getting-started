= Module 1: Deploy the App -- 3 Ways to Get Code Running
:source-highlighter: rouge
:toc: macro
:toclevels: 2

There is no single right way to get an application onto OpenShift -- and this lab proves it.
You'll deploy the DataSphere application three different ways: from a pre-built container image,
from source code using Source-to-Image, and as a complete multi-component Helm chart. Along
the way you'll learn the three building blocks that make every OpenShift application work: *Pods*,
*Services*, and *Routes*.

By the end, you'll have a fully working DataSphere instance -- UI, API, and database --
deployed from a single Helm chart, and you'll understand exactly what each piece does.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Identify and describe the role of Pods, Services, and Routes in OpenShift
* Deploy a containerized application from a pre-built image using the `oc` CLI or the web console
* Build and deploy an application from source code using Source-to-Image (S2I)
* Deploy a multi-component application in a single command using a Helm chart
* Explain when to reach for each of the three deployment methods

=== Step 1.3: Explore the Pod

A *Pod* is the smallest deployable unit in OpenShift. It wraps one or more containers and gives
them a shared network and storage context. Pods are ephemeral -- they can be created, destroyed,
and replaced at any time. A *Deployment* manages this lifecycle and ensures the right number of
pods are always running.

[%collapsible]
.Using the Terminal (CLI)
====
Inspect the frontend pod in detail. The `-l component=ui` flag selects pods by label:

[source,role="execute"]
----
oc describe pod -l component=ui
----

[%collapsible]
.Expected output (key fields)
=====
[subs="attributes+"]
----
Name:         datasphere-ui-597f67c7bc-xxxxx
Namespace:    {user}-datasphere
Labels:       app=datasphere
              component=ui
Containers:
  ui:
    Image:    {datasphere_ui_image}
    Port:     8080/TCP
    State:    Running
    Ready:    True
----
=====

Notice the *Labels*: `app=datasphere` and `component=ui`. These are how OpenShift connects
resources together -- Services find their Pods by matching these labels.

Notice the *Selector* field: `app=datasphere,component=ui`. This label-based selection is how

=== How they connect

[mermaid]
....
flowchart TB
    B["Browser"]
    R["Route\n(datasphere-ui)"]
    S["Service\n(stable internal IP)"]
    P1["Pod\n(datasphere-ui-n7rlv)"]
    P2["Pod\n(datasphere-ui-fgh12)"]:::dashed
    P3["Pod\n(datasphere-ui-zxv9k)"]:::dashed

    B -->|"HTTPS"| R
    R --> S
    S -->|"label selector"| P1
    S -->|"label selector"| P2
    S -->|"label selector"| P3

    classDef dashed stroke-dasharray: 5 5
....

The Route knows which Service to send traffic to.
The Service knows which Pods to forward requests to via label selectors.
The Pod runs your container.

=== Deploy the API via Source-to-Image

Rather than deploying from a pre-built image, you can point OpenShift directly at a source
code repository. OpenShift detects the language, installs dependencies, builds a container

This is *Source-to-Image* (S2I).

Add a backend, the frontend connects to it. That's it.

== Section 3: Deploy everything with a Helm chart (6 minutes)

You deployed DataSphere in two stages: the UI from a container image, the API from source.
That works, but in practice most teams don't run individual commands one at a time.
They package their application as a *Helm chart* -- a single installable unit that deploys
every component together.

A chart is a directory of templated Kubernetes manifests plus a `values.yaml` file that controls
configurable inputs (image tags, theme, replica count, and so on). When you run `helm install`,
Helm renders those templates and submits all the resulting manifests to the cluster in one shot.
It also records what it installed as a named *release*, so it can upgrade, roll back, or cleanly
remove every resource it owns.

== Module summary

You deployed the DataSphere application three different ways and learned the building blocks
behind every OpenShift application.

*The three building blocks:*

* *Pod*: the running container; ephemeral and managed by a Deployment
* *Service*: stable internal network address; uses label selectors to find pods
* *Route*: external access with a public hostname and automatic TLS termination

*The three deployment methods:*

* *Container image*: fastest path when the image is already built and published;
  no build step, no source code required
* *Source-to-Image*: point OpenShift at a Git repository and it builds, packages,
  and deploys the application; no Dockerfile, no local tooling required
* *Helm chart*: package the entire multi-component application for repeatable,
  version-controlled deployments; install or remove the whole stack in one command

keeps your application available automatically with scaling and self-healing.
