= Module 3: Git Is Your Source of Truth
:source-highlighter: rouge
:toc: macro
:toclevels: 2

In Modules 1 and 2, you deployed and operated DataSphere by running commands.
Those commands work -- but they leave no record of what changed, when, or why.
If a colleague updates something in the cluster, how do you know? If something
breaks overnight, how do you roll it back?

GitOps answers those questions by making a Git repository the source of truth
for the cluster. You declare what you want in Git; a controller called ArgoCD
watches that repository and keeps the cluster in sync. Every change is a commit.
Every rollback is a revert. The audit trail is the commit history.

In this lab, you'll deploy DataSphere using GitOps, make a live configuration
change by editing a file in Gitea, and watch ArgoCD apply it to the cluster
automatically -- no `oc apply`, no SSH, no deployment scripts.

toc::[]

== Learning objectives

By the end of this lab, you'll be able to:

* Explain what GitOps is and why Git is used as the source of truth for cluster state
* Register an application in ArgoCD by pointing it at a Git repository
* Make a configuration change through Git and watch ArgoCD apply it automatically
* Explain what drift detection is and why it matters

== The application: DataSphere

DataSphere is a datacenter status dashboard that simulates a global network of compute sites.
You can toggle sites online or offline, adjust simulated traffic load with a slider, and watch
the map update in real time. The application runs as three components:

* *datasphere-ui* — a React frontend served by nginx that renders the world map
* *datasphere-api* — a Python FastAPI backend that tracks datacenter state and load
* *datasphere-redis* — a Redis instance that keeps state consistent across API replicas

A pre-deployed DataSphere instance is already running in your `{user}-datasphere` project.
Open it now to see what you'll be deploying and managing in this lab.

== Section 1: Your ArgoCD instance (3 minutes)

=== Step 1.1: Confirm your ArgoCD instance is empty

After logging in, you'll see the ArgoCD dashboard with no applications listed.
That's expected -- your ArgoCD instance starts empty. You'll register the first
application in Section 2.

ArgoCD is a *GitOps controller*: a process that runs inside the cluster, watches
one or more Git repositories, and continuously reconciles the cluster state to match
what's declared in Git. When you tell ArgoCD about a Git repository, it creates an
*Application* -- a link between a Git source and a cluster destination. ArgoCD polls
that source on a regular interval and applies any changes it finds.

image::lab-03-01-argocd-empty.png[ArgoCD dashboard showing no applications]

Your ArgoCD instance manages only your own namespaces -- it cannot see or affect
other students' resources. Let's give it something to manage.

=== Step 3.2: Edit values.yaml

Click `values.yaml` in the file list.

image::lab-03-09-gitea-values-yaml-edit.png[Gitea file view showing values.yaml with the pencil edit icon in the top-right corner]

Click the *pencil icon* (Edit this file) in the top-right corner of the file view.

Find the `extraDatacenters: []` line in the `api:` section.

Replace `extraDatacenters: []` with the following block.

IMPORTANT: Use *2 spaces* per indent level -- not tabs. The Gitea editor may default to
4-space tabs, so type the spaces manually or adjust the editor settings. YAML will break
if the indentation is wrong.

[source,yaml,role="copypaste"]
----
  extraDatacenters:
    - id: "rh-mea"
      display_name: "MEA Hub"
      city: "Nairobi"
      region: "Middle East & Africa"
      tier: "major"
      lat: -1.3
      lng: 36.8
      base_capacity_pct: 45
      base_workload_count: 892
----

The `extraDatacenters` key must be indented 2 spaces under `api:` to match the
rest of the file.

=== What just happened

When you committed `values.yaml`, Git recorded the change with your username,
a timestamp, and the commit message you wrote. ArgoCD detected the new commit,
re-rendered the Helm chart with the updated values, and applied only the resources
that changed -- in this case, the `datasphere-config` ConfigMap. The API pod
restarted to pick up the new ConfigMap contents.

Every step is traceable. If you need to roll back, you revert the commit -- ArgoCD
applies the previous state. No manual intervention, no "I remember running some
command last week."

== Section 4: What happens when someone bypasses Git (5 minutes)

GitOps isn't just about deploying through Git -- it's about *only* deploying
through Git. The reason Self Heal exists is that sometimes people make changes
directly in the cluster: a quick `oc scale` to test something, a manual ConfigMap
edit under pressure. With Self Heal enabled, ArgoCD detects and corrects those
changes automatically.

Let's see it.

=== Step 4.1: Make a direct change to the cluster

Scale the API deployment down to 0 replicas directly, bypassing Git:

[source,role="execute",subs="attributes+"]
----
oc scale deployment datasphere-api --replicas=0 -n {user}-gitops
----

[%collapsible]
.Expected output
====
----
deployment.apps/datasphere-api scaled
----
====

=== Step 4.2: Watch ArgoCD respond

Switch back to the *ArgoCD* tab. ArgoCD continuously compares the cluster state
against Git -- with Self Heal enabled, it checks frequently and will detect the
scaling change on its own. If the application is not already showing *OutOfSync*,
click *Refresh* to trigger an immediate check.

The status shows *OutOfSync*: the cluster has 0 API replicas, but
`values.yaml` in Git says 1. ArgoCD knows what Git declared, and it knows what the
cluster has -- and those don't match.

Because *Self Heal* is enabled, ArgoCD restores the API to 1 replica automatically
within the next sync cycle. Optionally, watch it come back using either the CLI
or the web console:

[%collapsible]
.Using the Terminal (CLI)
====
[source,role="execute",subs="attributes+"]
----
oc get pods -n {user}-gitops
----

[%collapsible]
.Expected output
=====
----
NAME                                READY   STATUS    RESTARTS   AGE
datasphere-api-7d464fc7c4-xxxxx     1/1     Running   0          15s
datasphere-redis-69c8864976-xxxxx   1/1     Running   0          15m
datasphere-ui-597f67c7bc-xxxxx      1/1     Running   0          15m
----
=====
====

[%collapsible]
.Using the Web Console
====
In the console, switch to the *{user}-gitops* project and click
*Workloads* → *Pods*. Watch the `datasphere-api` pod return to *Running* status.

image::lab-03-12-console-api-pod-restored.png[Pods list showing datasphere-api back in Running status after ArgoCD Self Heal restored it]
====

The API pod is back. You didn't touch it -- ArgoCD matched the cluster to Git.

If you want to scale the API to 0 permanently, there's one way to do it: commit
a change to `values.yaml` that sets `api.replicas: 0`. Then Git reflects your
intent, and the cluster follows.

== What you learned

* *GitOps* means Git is the system of record for what runs in the cluster.
  You declare the desired state in a repository; a controller reconciles the cluster to match.
* *ArgoCD* is that controller. It watches a Git repository and applies changes automatically.
* *Drift detection* catches changes made outside of Git and corrects them -- either
  automatically (Self Heal) or by alerting you and waiting for a manual sync.
* Every configuration change is a Git commit: auditable, reversible, and reviewable.

The DataSphere app you deployed in Modules 1 and 2 used `oc` commands and Helm
from your terminal. The version you deployed in this lab came entirely from a
Gitea repository. Both work -- but only the second one has a history.
